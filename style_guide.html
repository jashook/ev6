Style Guide

File System:

Folders:

All code is split up into a minimum of two folders.

    1) include - Contains all class declarations and modules (see module definition)

    2) src - includes all code that is specific to that project that is not a module (e.g. main)

Other folders are added as needed to keep related items together.

    For example, the include folder may have subfolders named msft and osx for os specific classes and modules

Filenames -

        1) All filenames are underscore_delimted

            e.g. hello_world.hpp

        2) .h and .hpp files 

            i) .h files - used with c language only

            ii) .hpp files - used with c++ only (h plus plus)

Makefiles:

Because these folders are independently made of an IDE, IDEs will not compile the code by default.  Custon makefiles are required to compile the code, a sample is provided below.  The makefile is named "makefile" in all undercase and can be ran using the make command.  NOTE, you need to break the style guide to properlly format a makefile.  They expect \t characters instead of spaces.

FILES := $(wildcard test_files/*.cpp)

UINCLUDE := untested INCLUDE := tested

debug: $(FILES) g++ -I $(UINCLUDE) -I $(INCLUDE) $(FILES) -lrt

debug-full: $(FILES) g++ -I $(UINCLUDE) -I $(INCLUDE) -g $(FILES) -lrt

release: $(FILES) g++ -I $(UINCLUDE) -I $(INCLUDE) -O2 $(FILES) -lrt

The most important thing to note about the makefile is there needs to be at least two different make targets, one debug and one release.  Also note that the -I option under g++ is used to mark an include location from a specific directory.

Variables Naming Convention:

The naming convention used is a combination of camelCase, underscore_delimited, hHungarianNotation and C++'s stl style guide.

camelCase - Every new word is capitalized

underscore_delimited - Every new word is seperated by an underscore

hHungarianStyle - Every word is capitalized like camelCase, however, the type of the variable is the first letter of the name.  For example, iSize would be a variable of type int (i) named size.

All of the naming conventions are used in specific places

    1) global versus private variables

        i) global variables - variables are written without a leading underscore with a prefexed g, uses underscore_delimited style

            e.g. int g_amount_of_people = 5; declared a global int variable, which is easily understood by the prefixed g

        ii) any scoped variable - written with a leading underscore to signify that they are private and should or cannot be used in a scope other than the one they are in, after the underscore the rest of the name is camelCase

            e.g. int _CurrentIndex; declared a locally scoped integer

        iii) public variables - written without a leading underscore

        iv) private variables - written with a leading underscore

    2) static and constant variables

        i) static variables - written without a leading underscore but prefixed with s, the rest of the variable is written with underscore_delimited style

            e.g. static int s_size; declares a static int named size

        ii) const variables - written in all caps, underscore_delimited

            e.g. static const int MAX_SIZE

    3) member variables and member functions

        i) member variables - written with a prefixed m, underscore delimited

            a) private member variable - private char _m_array[20]

            b) public member varialbe - public char m_array[20]

        ii) member functions - underscore_delimeted and follows public and private scoping rules

            a) private member function - private void _ctor()

            b) public member function - public int get_size()

    4) template variables - prefixed with two underscores

            e.g. template<typename __Type>

define and #ifdef:

 1) #ifndef - when defining a file in C++ to avoid multiple includes, prefix and postfix two underscores to the file name and use underscore delimiting

    e.g. #ifndef __HELLO_WORLD_HPP__

Namespaces:

Do not use using unless you are repeatidly calling one function hundreds of times in the same file.

    e.g. if you will be using cout hundreds of times, you are permitted but discouraged to do the following -

        using namespace std;

    preferred is std::cout

All code written in C++ is done in its own namespace.  End of namespace has the namespace name in parentheses

    e.g.

        namespace ev7n {

            ...code...

    } // end of namespace(ev7n)

Comments:

1) C vs C++ comments

    C language does not recognize C++ comments or //.  Therefore, in order to not confuse languages use C++ style comments in C++ only, do not use
    C language comments or /*  */ in C++ unless you absolutely need to comment multiple lines, if you do that, change the comments to C++ comments upon release

2) Header Comments

    80 characters long, two lines, then skip a line, add author, skip another line then add the Module name.  The module name is the name of the file, skip another line and add modifications.  Modifications grows down, therefore, the last line in the modifcations should be the date it was created.  Modifications follow the following style, DATE (dd-MM-yy): Version number: What was changed.  Skip another line and end with two more 80 character lines of comments.

        e.g.

        ////////////////////////////////////////////////////////////////////////////////
        ////////////////////////////////////////////////////////////////////////////////
        //
        // Author: Jarret Shook
        //
        // Module: avl_tree_iter.hpp
        //
        // Modifications: 
        //
        // 21-Jan-13: Version 1.1: Updated to namespace ev6
        // 1-Jan-13 : Version 1.0: Created
        //
        // Notes:
        //
        // OPTIONAL NOTE SECTION
        //
        ////////////////////////////////////////////////////////////////////////////////
        ////////////////////////////////////////////////////////////////////////////////

3) Section Comments

    To divide important sections include two lines of 80 character comments

        e.g. /* ************************************************************************** */
             /* ************************************************************************** */

            See the sample file for more examples

4) Comments as explination

    The focus of this style is to be able to write code that is very readible, following the approach that you do not need very many comments if you have well named variables and functions.

        e.g. 

            def create_string_list(_IntegerList):

                return [_StringRepresentation for _StringRepresentation in _IntegerList]

    For example, without knowing python or its list comprehension syntax it is possible to understand from the function and variable names that the function returns a string list and takes an integer list as a parameter.

    There are time however when comments are necessary even if you avidly in support of readible variables in place of comments, most common this occurs to make things you cannot control more readible

        e.g. 

            _Size = 1000000 # 1,000,000

    If you must use comments to explain code use the following style

        /////////////////////////////////////////////////////
        // WIN32 Specific functions
        /////////////////////////////////////////////////////

    The comments should stop at either character count 50 or 60, whichever looks better

5) Comments to explain functions

    Functions that are outside of classes should have a header that is exactly as follows

    ;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;
    ;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;
    ;
    ; Function: insert_tree
    ;
    ; Parameters:
    ;
    ;     EAX: pointer to a tree structure
    ;     ECX: string to be inserted
    ;
    ; Uses:
    ;
    ;     EAX
    ;     EBX
    ;     ECX
    ;
    ; Returns:
    ;
    ;     VOID
    ;
    ;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;
    ;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;

    Uses in the example is assembly specific

Templates:

C++ specific, all template declarations occur on the same line as the class or function and use typename not the class keyword

    e.g. template<typename __Type> class vector

Whitespace:

NEVER USE TABS, insert spaces for tabs, 3 spaces = 1 tab, unless using html in which case 2 spaces = 1 tab

Use whitespace between lines, however, unless in python do not add whitespace on the first line of a new scope

    e.g.

        int main()
        {
            std::cout << "Hello World" << std::endl; // do not add whitespace

            return 0; // whitespace added

        } // whitespace added

Brackets:

Brackets always appear below the function or class declaration

e.g.

    int main() 
    {

    }
